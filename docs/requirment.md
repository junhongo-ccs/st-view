# 船舶リアルタイム可視化デモダッシュボード 要件定義

## 1. 目的
顧客向け提案デモとして、船舶位置確認を単一 Web 画面でリアルタイム可視化し、業務効率化の将来像を直感的に提示する。

## 2. システム要件
### 2.1 使用技術
- フロントエンド: Streamlit
- バックエンド: Python 3.11+
- 地理情報処理: GeoPandas, Shapely
- データ分析: Pandas
- 地図表示: Folium + streamlit-folium（第一優先）
- 設定管理: python-dotenv

### 2.2 データ取得
- 優先: AISstream.io WebSocket API
- 本PoCでは AISstream.io WebSocket API のみを利用する

### 2.3 無料PoC向けAPI候補
- AISStream
  - 方式: WebSocket（リアルタイム）
  - 特徴: 公式説明として無料のリアルタイム提供が明示される
  - 注意点: ブラウザ直叩きは避け、バックエンドでWebSocket受信してフロントへ中継する構成を推奨
- MyShipTracking API（Trial）
  - 方式: REST API
  - 特徴: 無料トライアルキー取得導線あり（Trial: 2000 coins、最大10日、90 calls/min）
  - 注意点: レート/期間制限を前提にPoC範囲を定義する
- AISHub
  - 方式: API（参加型データ共有）
  - 特徴: 自身のAIS受信データ共有を前提に、集約フィードへのアクセスを得るモデル
  - 注意点: 受信設備やデータ提供前提があるため、短期PoCの初手には不向き

### 2.4 PoC採用方針
- 第一候補: AISStream（リアルタイム性と導入容易性のバランスが良い）
- MyShipTracking / AISHub は比較検討情報として保持し、現行PoC実装対象からは除外する

## 3. 画面要件
### 3.1 レイアウト
- Streamlit ワイドレイアウト
- 構成: KPI エリア / サイドバー / 地図 / テーブル

### 3.2 KPI
- Tracked Vessels
- Under Way
- At Anchor
- ETA Today（未取得時はダミー値許容）

### 3.3 サイドバー
- 海域選択（東京湾、大阪湾、伊勢湾、瀬戸内海、九州北部）
- 自動更新 ON/OFF
- 更新間隔（30, 60, 120秒）
- フィルター（MMSI、国籍、船名部分一致）
- Future Vision パネル

### 3.4 地図
- 初期海域: 東京湾
- 初期ズーム: 8
- 表示項目: latitude, longitude, mmsi, ship_name, speed, course, status, destination, eta
- Tooltip: 船名 + 速力
- Popup: 船舶詳細
- 色分け: 0〜1kt 赤、1〜10kt 黄、10kt超 緑

### 3.5 テーブル
- 列: Timestamp, Ship Name, MMSI, Latitude, Longitude, Speed, Course, Status, Destination, ETA
- 機能: 並び替え、検索、CSV ダウンロード

## 4. データ処理要件
- 受信 JSON を正規化し DataFrame 化
- GeoDataFrame へ変換
- 最新 500 件を保持し古いデータを削除

## 5. 更新要件
- 30秒を基準とした定期更新
- streamlit-autorefresh または rerun で再描画

## 6. エラーハンドリング
- API接続失敗: AISデータへ接続できません
- データ未取得: 現在表示可能な船舶データが存在しません
- API制限: 無料API利用中のため一部データは取得できません

## 7. デモ演出要件
- ヘッダーに顧客向けメッセージ表示
- Future Vision をサイドバーに明示
  - 衛星AIS連携
  - 世界中の船舶追跡
  - 入港遅延予測
  - 過去航跡分析
  - サプライチェーン可視化
  - アラート通知

## 10. 受入条件
- ローカル環境で起動できる
- ダミーデータモードで全画面要素が表示される
- API モードでデータ更新される
- 接続失敗時も画面が停止しない
- CSV エクスポートが動作する

## 11. UIデザイン適用要件（design.md 反映）
本要件は [docs/design.md](docs/design.md) の業務アプリ向けデザインルールを、本ダッシュボード実装に適用したものである。

### 11.1 画面前提
- 対象: PC 専用（基準 1920x1080）
- Streamlit ワイドレイアウトを前提に情報密度を優先
- スマートフォン最適化は必須要件外

### 11.2 レイアウト密度
- 画面外側余白は 24px 相当を基準
- パネル間余白は 16px 相当を基準
- パネル内余白は 16px（高密度箇所は 12px）を基準
- KPI、地図、テーブルを 1 画面内で視認可能に配置

### 11.3 タイポグラフィ
- 見出し階層を明確化し、本文は過大サイズを避ける
- 標準本文は 14px 相当、補足は 13px、ラベルは 12px を基準
- テーブル本文は 13px 相当を基準
- 過剰な大見出し（text-3xl 以上相当）は使用しない

### 11.4 色とコントラスト
- 背景は淡色、本文は濃色の業務UI向け配色を採用
- 強調色は主要アクションと重要指標に限定
- 警告・エラー・成功は状態表示用途のみに限定
- 船舶速度色分けは既定どおり維持（赤/黄/緑）

### 11.5 パネルと装飾
- カードは情報グルーピング目的で使用し、過剰な装飾は避ける
- 角丸は中程度、影は最小限を原則とする
- 罫線で構造を示し、強い浮遊感を与えるデザインは採用しない

### 11.6 フォームと操作部
- 入力要素の高さを揃え、ラベルは入力欄上部に配置
- 主要操作ボタンは 1〜2 個を明確化
- 破壊的操作は警告色で明示

### 11.7 テーブル表示規約
- ヘッダーと本文の視認性を分離し、過度な行高は避ける
- 罫線は薄く、ホバー時に行追従しやすい見た目を採用
- 1 セル内の情報量を制御し、可読性を優先

### 11.8 明示的禁止事項
- 大きすぎる余白の多用
- LP 風の中央寄せレイアウト
- 本文での過剰な行間
- 過剰な角丸・濃い影の多用
- 情報量を下げる装飾優先のカード乱立

### 11.9 実装メモ（Streamlit）
- Streamlit は Tailwind クラスを直接利用しないため、同等の見た目ルールを CSS とコンポーネント構成で再現する
- 必要に応じて `st.markdown(..., unsafe_allow_html=True)` で最小限のスタイルを適用する
- 既存スタイル定義（リンク色、表罫線、ヘッダー背景）と競合しないよう段階的に適用する

## 12. UI受入条件（追加）
- KPI・地図・テーブルが 1920x1080 で過度なスクロールなく閲覧できる
- 見出し/本文/補足/ラベルの階層が視覚的に判別できる
- テーブルの可読性（ヘッダー識別、行追従、列視認）が担保される
- エラー/警告メッセージが状態色で判別できる

## 13. AISStream実装 最短チェックリスト（PoC）
本PoCは AISStream のみを対象とし、以下を「最低限の完成条件」とする。

### 13.1 接続・購読
- `.env` に `AISSTREAM_API_KEY` が設定されている
- バックエンドで WebSocket 接続を確立できる
- 対象海域（初期: 東京湾）の購読条件を送信できる
- 接続状態（接続中/切断/再接続中）をアプリ内で表示できる

### 13.2 データ処理
- 受信 JSON を正規化して DataFrame 化できる
- 最低限の表示項目（緯度/経度/MMSI/船名/速度/針路/状態）を抽出できる
- 同一船舶の重複データを抑制できる
- `MAX_RECORDS` 上限で古いデータを削除できる

### 13.3 可視化
- 地図上に船舶マーカーを描画できる
- 速度帯（赤/黄/緑）で色分けできる
- テーブルで検索・並び替え・CSV出力ができる
- KPI（Tracked/Under Way/At Anchor）を更新できる

### 13.4 障害耐性
- 切断時に自動再接続を試行できる
- API未取得時でも画面全体が停止しない
- 警告メッセージを表示し、操作継続できる

### 13.5 デモ運用
- `DATA_MODE=DUMMY` と `DATA_MODE=API` を切り替えて動作確認できる
- APIキー未設定時の案内メッセージが表示される
- 更新間隔（30/60/120秒）変更が画面更新に反映される

### 13.6 最低受入基準（Go/No-Go）
- 30分以上の連続稼働でアプリがクラッシュしない
- 再接続が 3 回以内の試行で復旧する
- 地図とテーブルが最新データへ追従する
- 接続失敗時でも手動更新/フィルタ操作が可能である
