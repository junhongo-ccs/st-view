【システム・ロジック要件】
1. URLパラメータ `?path=enc:<エンコードされたPolyline文字列>` を監視・読み取ること。
2. Google Maps APIの `geometry.encoding.decodePath()` を用いて、エンコード文字列を緯度経度（LatLng）の配列にデコードすること。
3. 取得した座標配列を等間隔（例: 5m）に間引き、Street View Static APIの画像URLリストを生成・プリフェッチ（事前読み込み）すること。
4. 画面の縦スクロール（`scrollTop`）を監視し、スクロール量に応じて配列のインデックスを算出。対応する画像（`<img>`のsrc）をクロスフェードで滑らかに切り替えること。

【UI/UX・スタイリング要件】
1. 極限までミニマルなデザインとクリーンなレイアウトを採用すること。ヘッダーやフッターは排除し、画像を画面全体（`object-fit: cover`）に配置する。
2. ストリートビュー特有のデフォルトUIは全て非表示にすること。
3. 画面右下に、細いアウトラインで描かれたシェア（リンク）アイコンをフローティング配置すること。
4. シェアアイコンをクリックすると、現在の経路情報を含んだURLを `navigator.clipboard.writeText` でクリップボードにコピーすること。
5. コピー成功時、画面中央またはアイコン付近に「Copied to clipboard」という小さなトースト通知をフェードインで表示し、約2秒後にフェードアウトさせること。
6. トースト通知やプログレスバーのアクセントカラーには、信頼感のあるディープブルー（例: #004386）を使用し、洗練されたプロフェッショナルな印象に仕上げること。

【スマホ最適化・フルスクリーン（没入感）要件】
ターゲットデバイスはiPhone 12〜最新のGoogle Pixel等のモダンなスマートフォンを想定し、ブラウザ特有のURLバーや下部メニューを感じさせないネイティブアプリのような挙動を実装してください。

1. **Dynamic Viewportの適用:** コンテンツの高さには `100vh` ではなく `100dvh` を使用し、ブラウザのUIの伸縮によってレイアウトが崩れたり見切れたりしないようにすること。
2. **Fullscreen APIの活用（Tap to Start）:**
   - 初回アクセス時は、画面のスクロールを無効化し、画面中央に「Tap to Start」というシンプルなテキスト（またはアイコン）を表示すること。
   - ユーザーが画面をタップした際、Fullscreen API (`requestFullscreen`) を呼び出してブラウザのURLバーと下部UIを非表示の全画面状態にすること。
   - 全画面化と同時に「Tap to Start」をフェードアウトさせ、ストリートビューのスクロール連動処理（Phase 3）を有効化すること。
3. **PWA / Standaloneモードへの対応:**
   HTMLの `<head>` 内に Web App Manifest (`manifest.json`) の読み込み設定、および iOS Safari向けの `<meta name="apple-mobile-web-app-capable" content="yes">` などのメタタグを含め、「ホーム画面に追加」された際に独立したアプリとしてURLバー無しで起動できるようにベースを整えること。
4. **オーバースクロールの防止:**
   CSSで `overscroll-behavior: none;` を `body` に指定し、画面端までスクロールした際のリロード（Pull-to-refresh）やバウンス効果を無効化すること。
この「オーバースクロールの防止（Pull-to-refreshの無効化）」も、WEBアプリ特有の安っぽさを消すための強力なスパイスになります。

この仕様なら、初回の「タップ」をトリガーにして一気に全画面の没入モードへ引き込めます。この方向でAIにコードを生成させてみましょうか？

1. **モバイル版（画面幅 768px未満）のUI:**
   - 画面上に検索入力フィールドは常設せず、ハンバーガーアイコン（または検索アイコン）を配置する。
   - アイコンタップ時、画面全体（または画面下部から）に検索専用のモーダルパネル（ドロワー）がスライドインするアニメーションを実装。
   - モーダル内で出発地・目的地を入力・選択し、「ルート生成」ボタンを押すとモーダルが閉じ、ストリートビューの再生待機状態（Tap to Start）へ遷移すること。

2. **PC版（画面幅 768px以上）のUI:**
   - 画面上部中央に、フローティング（浮遊）するカプセル状の検索トリガー（例：現在設定されているルートのサマリーテキストと検索アイコン）を配置する。
   - トリガーをクリックするとパネルが下方向へエキスパンド（展開）し、出発地・目的地の入力フィールドが出現する。
   - ルート生成の実行時、またはパネル外のオーバーレイをクリックした際は、スムーズなアニメーションと共に再び元のカプセル状に折り畳まれること（Airbnbの検索バーのようなアコーディオン挙動）。

スマホ版：ハンバーガーメニュー ＋ 検索ドロワー（表記揺れ対応）
スマホの極小画面では、メイン画面に検索窓を常設すると景観の邪魔になります。ハンバーガーメニューから「検索専用のレイヤー」を被せる手法は非常に効果的です。

UIの挙動:

画面の端（親指が届きやすい右下、または慣れ親しんだ左上）にハンバーガーアイコンを透過気味に配置。

タップすると、下から上に全画面の検索パネル（Bottom Sheet / フルスクリーンモーダル）がスライドイン。

表記揺れ・サジェストの解決策（Places API）:

Google Maps の Places Autocomplete API を組み込みます。これを使えば、「じゆうがおか」「自由ケ丘」「Jiyugaoka」といった表記揺れや、「スタバ」などの略称でも、サジェスト候補として正式名称と住所をドロップダウンで即座に提示してくれます。ユーザーはリストから「選ぶだけ」になり、入力ストレスが激減します。

PC版：フローティング・ヘッダー ＋ エキスパンド検索
PC画面では、上部にベタ塗りのヘッダーバーを固定してしまうと「業務システム感」が出てしまい、シネマティックな没入感が削がれます。ご提案の通り「入力時だけ展開する」のがベストです。

UIの挙動（フローティング・ピル型）:

Idle（待機）状態: 画面上部の中央に、丸みを帯びた小さなカプセル型のボタン（例：検索マーク 自由が丘駅 → STAMP COFFEE）だけがフワッと浮いています。

Active（入力）状態: カプセルをクリックすると、スッと下に広がり（エキスパンド）、しっかりとした「出発地」「目的地」の入力フィールドが現れます。

トランジション: 検索を実行するか、画面の余白（外側）をクリックすると、再び小さなカプセル型に折り畳まれます。AirbnbのPC版サイトの検索窓のような、リッチで洗練されたマイクロインタラクションです。

   ▼
【Phase 0: URLパラメータの判定と初期化】 (フロントエンド・ルーティング)
   │
   ├─ パラメータあり（例: `?start=lat,lng&end=lat,lng` または `?route_id=xyz`）
   │    └─ 入力UIをスキップし、パラメータの座標を元に 【Phase 1】へ直行
   │
   └─ パラメータなし
        └─ 出発地と目的地のセットを入力するUIを表示 ─(決定)→ 【Phase 1】へ
   │
   ▼
【Phase 1: ルートデータの取得と整形】 (Google Maps Directions API)
   │
   ├─ 1. APIからA地点〜B地点の「Polyline（経路の線データ）」を取得
   ├─ 2. Polylineをデコードし、ルート上の緯度経度（LatLng）を抽出
   └─ 3. 指定した間隔（例: 5メートル毎）で座標を間引き、均等な距離の【座標配列】を生成
   │
   ▼
【Phase 2: 画像の取得とキャッシュ】 (Street View Static API)
   │
   ├─ 1. 【座標配列】の各LatLngに対してAPIリクエストを生成（進行方向のHeadingを指定）
   └─ 2. スクロール時の遅延を防ぐため、画像のプリフェッチ（事前読み込み）を実行
   │
   ▼
【Phase 3: スクロール連動処理】 (フロントエンド / イベントリスナー)
   │
   ├─ 1. ユーザーの縦スクロール量（scrollTop）を監視
   └─ 2. スクロール位置と【座標配列】のインデックスをマッピング
   │
   ▼
【Phase 4: UI描画・トランジションと「共有」】 (DOM更新 / Clipboard API)
   │
   ├─ 景色のアクション: スクロールに応じて `<img>` のsrcをクロスフェードで切り替え
   │
   └─ 共有のアクション: 画面隅の「Share（またはリンクアイコン）」ボタン
        │
